- Since our main goal is fixing meal decision fatigue, we need the app to filter recipes instantly. Postgres is way better at handling all the relational math for ingredient scaling and complex dietary filters.
- It automatically spins up secure REST and GraphQL APIs for us, which saves us weeks of writing custom backend routing code.
- Also, mobile connections drop all the time when people are moving around. Supabase has this thing called Supavisor for connection pooling, which will stop our database from crashing if people's cell service cuts out while they're cooking.	- It's super startup-friendly. The free tier will easily cover us through the whole 6-week MVP build and QA testing phase. When we do need to upgrade, the Pro plan is pretty predictable, unlike AWS where a random viral recipe could unexpectedly cost us a fortune in compute time.	- Low to Medium. We can use the Supabase CLI to automatically generate strict TypeScript types for our frontend. This is awesome because if someone changes a database column, the React Native app will just fail to compile locally instead of breaking for users in production.
- Plus, we are plugging Directus (our CMS) straight into the same Postgres database, so we don't have to write any messy glue code to sync content between two completely different systems.
	- Security (Row Level Security): Since the mobile app talks directly to the database, our Row Level Security (RLS) rules have to be flawless. If we mess up the config, someone could just easily scrape our entire recipe library or see other users' private info.
- Directus Oopsies: Because the CMS connects directly to our production data, a non-technical content manager could accidentally change a database column type or delete a constraint from the admin panel and completely crash the app. We need to lock down their permissions hard so they can only touch recipe text and images.
	- MVP (6-Week Launch): We strictly need to get the core tables set up (users, recipes, ingredients) and heavily index them so the dietary filters feel instant. We also absolutely need to lock down the RLS policies before we launch.
- Later Phase (Phase 2+): Once we start selling DTC cubes, we'll need to hook this up to Shopify. I think we can use Supabase Edge Functions to listen for events and quietly sync our user profiles over to Shopify's customer records in the background, which will set us up perfectly for the rewards program down the line.

Plus, if Shopify's API goes down during Phase 1, no one can even get into the app.
    • ​Firebase Auth / Auth0: These are the industry standards, but using them with our Supabase PostgreSQL database feels super messy. We'd have to write custom webhooks just to sync Firebase user IDs with our own database, which just adds lag and more stuff that can break.
    • ​Supabase Auth (Integrated): This uses GoTrue and is built right into our database. When someone logs in, their data is instantly tied into our Row Level Security (RLS) rules without us having to write any extra glue code.	To hit our goal of "Zero Friction Paths", I really think we should use native Apple and Google sign-ins, plus passwordless magic links. Users will just close the app if we make them type out a password while their hands are messy from cooking.  
​Schema Architecture & Relational Mapping: Supabase keeps the auth stuff separate from the public app data. To make sure our CRM and personalization work smoothly, here's how I think we should set up the tables:
​auth.users (Managed by Supabase): This is a hidden system table that handles all the OAuth handshakes, password hashing, and sessions automatically.
​public.users Table (The Core Identity): Needs an id (UUID), email, and created_at timestamp. We should link the id directly to auth.users(id) as a Foreign Key with an ON DELETE CASCADE rule. We can write a quick PostgreSQL trigger to automatically drop a row in here the exact second someone finishes signing up.  
​public.user_profiles Table (The CRM Engine): We'll use this for the user_id, dietary_preferences (we can just use a Postgres array for things like vegan or high-protein), discovery_channel for marketing attribution, and the sauce_log to track what cubes they actually own. Keeping this separate from the core users table means we won't lock up the main database when people are constantly updating their profiles.  	​It's basically free for us right now. Supabase Auth gives us up to 50,000 Monthly Active Users (MAU) on the free tier, and the user schema just uses our regular PostgreSQL storage limits.	Low to Medium. Expo already has really good libraries for handling the Apple/Google logins on the frontend, so we don't have to deal with messy web-views. The only slightly tricky part for the backend is writing that first PostgreSQL trigger to safely sync auth.users with public.users when they first join, but it shouldn't be too bad.	​Insecure Token Storage: If we just dump session tokens into plain text AsyncStorage, someone could easily hijack a session. It's super important that we strictly enforce using expo-secure-store to lock the tokens in the iOS Keychain and Android Keystore.  
​RLS Data Leakage: Since user profiles hold specific dietary preferences, we need really strict Row Level Security (RLS) on both the public.users and public.user_profiles tables. We have to make sure users can only query or edit rows that match their exact auth token so nobody scrapes our user data.  	MVP (6-Week Launch): We absolutely need the smooth onboarding via Apple, Google, and Magic Links working flawlessly. ​Later Phase (Phase 2+): Later on, we can add fields for birthday and retail_location_preferences for the rewards program. The coolest part is we'll add a shopify_customer_id to the users table. Freya	Recipe Data Schema	- Document Store (NoSQL / MongoDB / Firestore): I thought about just dumping the whole recipe—ingredients, steps, metadata, and all—into a single massive NoSQL document because it's super easy to read on the frontend. But honestly, if we ever need to update a specific cube's branding across all recipes, or do crazy filtering like "give me all vegan recipes under 15 mins," a document store is going to choke. We'd have to write really messy batch updates and duplicate a ton of data.
- Graph Database (Neo4j): I looked at graph databases because they're really cool for mapping out how ingredients relate to each other, but it's way overkill for our Phase 1 MVP. We'd have to learn a whole new query language (Cypher) and it would just slow us down.
- Relational Database (PostgreSQL): This forces us to be super strict with our data shape, which is exactly what we need to make sure the macros and serving sizes scale perfectly on the phone without breaking the app.
	Relational schema via PostgreSQL (Supabase).
- To hit our "Speed to Meal" goal (finding dinner in under 30 seconds), we need super fast filtering. By splitting everything up into strict recipes, recipe_ingredients, and recipe_steps tables, we can handle the math for the dynamic serving size adjuster natively on the backend.
- We can use Postgres's built-in array types for the cube_tags[] and dietary_tags[]. If we slap a GIN index on those arrays, our app's search filters will feel lightning-fast even when people cross-reference a bunch of dietary tags at once.
	- It's totally included in our existing Supabase database plan, so building out a complex schema doesn't add any extra infrastructure costs to our startup budget.
	- Medium. It's a bit tricky because we have to set up really strict foreign key rules. Like, if a content manager deletes a draft recipe, we need an ON DELETE CASCADE rule to automatically wipe out the leftover ingredients and steps so our database doesn't turn into a messy swamp of orphaned rows.
- Also, we have to use strict decimal data types for the ingredient quantities instead of normal floating-point numbers. Otherwise, the math gets weird (like showing 0.300000004 cups instead of 0.3) and it'll completely break the frontend UI.
	- The N+1 Query Problem: If the mobile app grabs a list of 10 recipes, and then does 10 separate database calls to fetch the ingredients for each individual one, the app is going to lag out hard. We have to make sure we use joined views or Supabase's GraphQL endpoints to grab the whole payload in one single network trip.
- Directus Data Types: Since the content team is typing this stuff directly into the Directus CMS, we need bulletproof database constraints. If someone accidentally types a word into the timer_seconds field instead of a number, the Guided Mode countdown timer will literally crash the app mid-cook. We have to lock down the data types at the database level.
	- MVP (6-Week Launch): We absolutely need these three core tables mapped out perfectly so the Quick View, Guided Mode, and dietary filters actually work. Getting the dynamic serving size math right is a massive blocker for launch.
- Later Phase (Phase 2+): Later on, we'll need to level up the schema for the "Sauce Log" feature. We'll have to build new tables that link the user's account directly to the physical cubes they have in their real-life pantry, so we can trigger low-stock push notifications and prompt them to buy more from our DTC store.

Freya 	Saved recipe state	- Redux / Global State (Redux Toolkit): I spent some time looking into Redux because it's what everyone talks about for React state management, and it's taught in almost every bootcamp. But honestly, it's a massive overkill for what we need right now. To just track if someone clicked a heart or bookmark icon, we'd have to write so much boilerplate code—actions, reducers, configuring the store, and dispatchers. And the biggest dealbreaker is that Redux only handles client-side state. It doesn't actually talk to the database out of the box. If we used Redux, we'd have to bolt on heavy middleware like Redux Thunk or Redux Saga just to manage the asynchronous API calls to Supabase. That means we'd be manually writing code to track isLoading, isSuccess, and isError for every single button press. We just don't have the time in a 6-week MVP to manage all that plumbing when we really only need to sync a few simple user actions to the cloud.
- Local Storage Only (AsyncStorage): We could just save their favorite recipes straight to the phone's local storage. But if they delete the app or log in on their iPad, all their saved dinners vanish. Plus, we need this engagement data on the backend to figure out which cubes are actually popular.
- TanStack Query (React Query) + Supabase: This is the sweet spot. It handles the API calls, caches the data locally so the app feels crazy fast, and syncs everything to our database behind the scenes.
	TanStack Query (React Query) syncing to Supabase.
- For our goal of zero-friction cooking, we can't have the UI freeze with a loading spinner while it waits for the server to confirm a "save." 
- React Query lets us use "optimistic updates". This means when a user taps the heart or bookmark, the button fills in instantly on their screen, and React Query quietly syncs that action to our Supabase database in the background.
- It also handles caching automatically, so if they close the app and open their "Saved" tab again, the recipe library loads instantly on repeat visits without having to fetch it all from the network again.
	- TanStack Query is completely free and open-source for our React Native frontend. The actual saved states just get stored as simple relational rows in our existing Supabase database, so it doesn't add any extra infrastructure costs.
	- Medium. React Query is super powerful, but configuring the cache invalidation is kind of a headache. If we don't set up our query keys right, a user might un-save a recipe, but it still shows up in their "Saved" tab because the local cache didn't clear out. We have to be really precise with how we structure the frontend state logic.
	- Network Drops (No Offline Mode): The PRD specifically says we aren't supporting offline use at launch. If someone saves a recipe while walking into a dead cell zone in the grocery store, the background sync will fail. We have to build really good API retry logic for transient network failures, and show a clear, friendly error message instead of letting the app silently crash or freeze on a blank screen.
	- MVP (6-Week Launch): It's a hard requirement that when someone taps save, the recipe shows up in their Saved tab within 1 second, and that state has to survive if they close the app or switch phones. The optimistic UI updating is mandatory for launch.
- Later Phase (Phase 2+): We'll start using all this saved recipe data to power the "Discovery Mode" algorithm. If the backend knows they save a lot of spicy, 15-minute chicken recipes, we can use that data to prioritize showing them similar cards in the swipe stack.

    • App Review rejection is common on first submission (broken links/placeholder content, inaccurate metadata, incomplete App Privacy disclosures) — budget for at least one resubmission cycle
    • Review time is typically 24–48 hours but can stretch around major Apple events — don't submit on the last possible day
    • App Privacy questionnaire must match what the Analytics integration actually collects, or it risks rejection
	MVP:
    • Enroll in Apple Developer Program (start immediately, week 1)
    • Create App Store Connect record and bundle ID
    • TestFlight internal beta → external beta
    • Submit via EAS Submit, leaving 1 buffer cycle for possible rejection
    • Target iOS (App Store) only, per Phase 1 priority
Later-Phase:
    • Android / Google Play submission
    • Fastlane automation, if we move to weekly release cadence post-launch
